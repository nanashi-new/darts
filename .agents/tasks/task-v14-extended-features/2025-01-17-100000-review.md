# Six-feature expansion for v1.4: analytics, import formats, reports, profiles, guide, UX automation

This change adds six major features in a single pass: an analytics service with tournament/player statistics and comparison views, CSV/JSON/clipboard import routes unified through a dispatch pipeline, a configurable report builder with a template database, multi-profile support with a registry and selector dialog, an interactive guided tour with searchable help, and UX automation (keyboard shortcuts, toast notifications, undo stack, drag-and-drop import, session filter persistence). The implementation touches 43 files and adds ~4,400 lines. All six features follow the same architectural pattern—service layer separated from UI—and reuse existing infrastructure (the xlsx import's `TableBlock`, the export service's PDF renderer, the settings persistence layer).

**Watch for:** The undo manager uses a module-level singleton holding closures that capture references indefinitely (confirmed). The `compare_players` analytics method executes 2 queries per player in a loop—an N+1 pattern that will degrade with large comparison sets (confirmed). The report builder calls a private method (`_write_fallback_pdf`) on the `ExportService` instance, coupling it to an implementation detail (confirmed). Profile deletion uses `shutil.rmtree` without any confirmation beyond the UI dialog, and there is no backup before destructive filesystem removal (confirmed).

## High-level view

The analytics service provides seven read-only aggregation methods over tournaments and player results. It's stateless and takes a connection per call, which keeps it testable, but the `compare_players` method pays for simplicity with an N+1 query pattern that issues two queries per player rather than batching.

The import pipeline extends the existing xlsx flow by adding CSV (with auto-detected delimiter), JSON (with field synonym mapping), and clipboard (tab-separated) parsers, all returning the same `TableBlock` dataclass the UI already knows how to handle. The unification is clean: `parse_tables_from_file` dispatches by extension, and the UI calls a single `_import_blocks` helper regardless of source.

The report builder assembles multi-section reports from SQL queries filtered by date, league, and category, then writes to text, xlsx, or PDF. Templates are stored in a new `report_templates` table with JSON config blobs. PDF output delegates to `ExportService._write_fallback_pdf`, a private method that renders a single-page latin-1-encoded PDF limited to 120 lines—Cyrillic text will degrade to replacement characters.

Multi-profile support introduces a `ProfileManager` that manages directories under a shared base, each containing its own `app.db` and `settings.json`. Profile switching updates a `profiles.json` registry and takes effect on next launch. The runtime path resolver now reads this registry to locate the active profile, adding a filesystem read on every cold start.

The guided tour is a simple step-through overlay that stores completion state in settings. The help view replaces the old FAQ tab with a searchable markdown viewer and a "start tour" button. Search is naive substring matching against markdown sections.

UX automation adds a `ShortcutManager` that registers global shortcuts on the main window, a `ToastNotification` label that auto-hides after 3 seconds, a 20-slot undo stack exposed as a module-level singleton, session filter persistence in the settings file, and drag-and-drop file import on both the main window and the import/export view.

1. **Undo singleton holds closures indefinitely** — The global `undo_manager` instance retains up to 20 closures that may capture large objects (DB connections, widget refs). Consider weak references or explicit `clear()` on profile switch.
2. **compare_players N+1 queries** — Issues 2 SQL queries per player ID. Batch into a single query with `WHERE player_id IN (...)` and group in Python.
3. **Report PDF Cyrillic corruption** — `_write_fallback_pdf` encodes the stream as latin-1 with `errors="replace"`, so all Cyrillic characters become `?`. Reports with Russian text will be unreadable in PDF format.
4. **Report builder couples to private API** — Calling `self._export._write_fallback_pdf(...)` binds to an implementation detail of `ExportService`. Promote to a public method or extract a PDF writer.
5. **Profile deletion has no undo/backup** — `shutil.rmtree` is irreversible. Consider creating a restore point or moving to trash before removing.
6. **Bare `except Exception` swallows errors silently in UI** — All analytics view load methods (`_load_stats`, `_load_progress`, `_compare`) catch `Exception` and silently pass. Failures are invisible to the user; at minimum log or show a toast.
7. **Drag-drop handler re-imports `detect_format`** — `handle_dropped_files` does a local import of `detect_format` and `parse_tables_from_file` from `import_pipeline` despite `parse_tables_from_file` already being imported at module level. The `detect_format` call is redundant since `parse_tables_from_file` dispatches internally.
8. **Guided tour always shows on first launch regardless of profile** — Completion is stored in the settings file, but if the user switches profiles (which have separate settings), the tour would replay. Consider storing completion at the user/system level rather than per-profile.

## Undo manager singleton and closure retention

The undo stack is a module-level global:

```python
# Global singleton instance
undo_manager = UndoManager()
```

Each pushed action stores an arbitrary `Callable[[], None]`. Because the stack retains up to 20 entries and there's no lifecycle hook to clear it (e.g., on profile switch or window close), captured objects in those closures—database rows, widget references, large data structures—remain alive for the process lifetime. In a desktop app that may run for hours, this is a slow resource leak. The `ShortcutManager` invokes `undo_manager.undo()` on Ctrl+Z and shows a toast, but nothing clears the stack when the user changes context or switches profiles.

The tests verify push/pop mechanics and the 20-item cap but don't exercise closure capture behavior or lifecycle cleanup.

## Import pipeline: clean reuse, redundant dispatch in drag-drop

The CSV, JSON, and clipboard parsers all produce `TableBlock` objects by reusing `detect_headers`, `validate_rows`, and `_calculate_mapping_stats` from `import_xlsx`. This means all format-specific parsing converges at the same data contract, and the UI's `_import_blocks` helper handles everything uniformly. The delimiter auto-detection in CSV (Sniffer with manual fallback) and the synonym mapping in JSON are well-designed for a multilingual user base.

However, `handle_dropped_files` re-imports and calls `detect_format` before calling `parse_tables_from_file`, which already performs the same dispatch internally:

```python
def handle_dropped_files(self, paths: list[str]) -> None:
    from app.services.import_pipeline import detect_format, parse_tables_from_file
    for path in paths:
        fmt = detect_format(path)
        if fmt in ("xlsx", "csv", "json"):
            blocks = parse_tables_from_file(path)
```

The `fmt in ("xlsx", "csv", "json")` guard is always true because `detect_format` only returns those three values—the check provides no filtering. This is dead code that could confuse future maintainers.

## Report PDF output: latin-1 encoding corrupts Cyrillic

The `_write_fallback_pdf` method encodes the content stream as `latin-1`:

```python
stream = "\n".join(stream_lines).encode("latin-1", errors="replace")
```

Since the app's content is overwhelmingly Russian (player names, tournament names, column headers are all in Cyrillic), every non-latin character becomes `?` in the PDF. The text and xlsx output paths handle this correctly, but the PDF path produces a corrupt document for any real data. This isn't new code—it's a pre-existing limitation of `ExportService`—but the report builder now routes through it as a first-class output format, making users more likely to hit it.

## Analytics compare_players: per-player query loop

```python
def compare_players(self, connection, player_ids):
    result = []
    for player_id in player_ids:
        player_row = connection.execute(
            "SELECT ... FROM players WHERE id = ?", (player_id,)
        ).fetchone()
        ...
        rows = connection.execute(
            "SELECT ... FROM results WHERE player_id = ?", (player_id,)
        ).fetchall()
```

For N players, this executes 2N queries. The ComparisonTab UI populates a multi-select list from which users can pick arbitrary numbers of players. With 10-20 players this is fine; with 50+ (plausible for "compare all players in a league") it becomes noticeably slow on SQLite. A single batched query with `GROUP BY player_id` would collapse this to 1-2 queries total.

## Profile lifecycle and deletion safety

`delete_profile` performs `shutil.rmtree(profile_path)` immediately when the user confirms in the dialog. There's no intermediate step—no trash/archive, no restore point creation, no undo action pushed to the undo stack. Given that a profile directory contains the entire `app.db` (all tournament data, player records, audit logs), this is a high-severity destructive operation with no recovery path.

The guard against deleting the *current* profile is sound (it checks `last_used` in the registry), but if the registry is corrupt or the path comparison fails due to symlinks, the guard could be bypassed.

## Silent error swallowing in analytics UI

Every data-loading method in `TournamentAnalyticsTab`, `PlayerAnalyticsTab`, and `ComparisonTab` wraps its logic in:

```python
try:
    ...
except Exception:  # noqa: BLE001
    pass
```

If the database connection is stale, the schema is missing columns, or any SQL error occurs, the user sees nothing—no error message, no empty-state label, no log entry. The `noqa: BLE001` annotation acknowledges this is a bare exception catch but doesn't fix the UX gap. At minimum, a toast notification or a status label would surface failures.

## Guided tour completion scope

Tour completion is stored via `update_setting(_SETTINGS_KEY, True)`, which persists to the current profile's `settings.json`. When a user creates a new profile and switches to it, the tour will show again because the new profile has its own settings. This might be intentional (introduce the app to each data context) or surprising (the user already knows the app). If intentional, it should be documented; if not, completion should be stored at a level above profiles.
