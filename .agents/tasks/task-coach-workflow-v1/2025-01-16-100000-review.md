# Coach Workflow v1 + UI/UX Customization Bundle

This diff delivers two large feature sets in a single branch: (1) a Coach Workflow system—coach_tasks and training_plans tables with services, a dedicated "Тренер" tab, a 7th "План тренера" tab in PlayerCardDialog, and an analytics dashboard; and (2) a UI/UX customization layer—theme engine, settings view, tags, attachments, custom fields, tournament list with filters, status stepper, and branding. The combined surface area is ~6,000 lines across 59 files.

**Watch for:**
- Dashboard makes 5+ separate DB round-trips per refresh with no caching or debounce; UI stalls are possible on large datasets (confirmed).
- Deletes of coach tasks and training plans produce no audit log entry, unlike every other mutation on these entities (confirmed).
- Audit event type naming uses UPPER_CASE for new coach events but lowercase for pre-existing tournament events—label lookup will fail silently for the new events if any display path uses the wrong case (confirmed).
- Attachment file copy stores absolute paths relative to profile_root but never cleans up the physical file on `delete_attachment`—orphan files accumulate (confirmed).

## High-level view

The data layer follows existing conventions cleanly: two new tables with CHECK constraints, typed repositories, frozen-dataclass service records, and parameterized SQL. The exercises column uses a TEXT/JSON approach that's pragmatic for a local desktop app. Foreign-key ON DELETE behavior is intentionally asymmetric—coach_tasks SET NULL on player delete, training_plans CASCADE—which mirrors the domain intent (general tasks survive player removal, plans don't).

The CoachView widget is the richest new UI surface. It holds its own connection, builds three sub-tabs, and refreshes all sub-tabs (including a 5-query dashboard) on every individual action. This is fine for a handful of records but the query fan-out grows linearly with the data, with no circuit-breaker.

The audit logging gap is limited to deletes: create, update, and complete all emit events, but `delete_coach_task` and `delete_training_plan` do not. This breaks the assumption that the audit log is a complete history of mutations.

The UI/UX customization (theme, tags, attachments, custom fields) appears in the same branch despite being a separate concern. It broadens the review surface but the implementation is self-contained and doesn't entangle with the coach workflow at the data layer.

(confirmed)

## Dashboard query fan-out in CoachView

`_refresh_dashboard` (called after every create, edit, delete, and complete action) makes these service calls sequentially:

```python
list_coach_tasks(connection=..., status="open")
list_coach_tasks(connection=..., status="in_progress")
list_overdue_tasks(connection=...)
list_coach_tasks(connection=..., priority="urgent")
list_coach_tasks(connection=..., status="done", include_done=True)
list_training_plans(connection=..., status="active")
```

Each call instantiates a new `CoachTaskRepository`, builds a query, executes it, and maps every row to a dataclass. The done-this-month logic then filters in Python by comparing `completed_at` to the first-of-month ISO string. For 50+ tasks this creates measurable UI lag on the main thread. The dashboard tab should either skip refresh when it's not the active sub-tab, or consolidate into a single SQL `GROUP BY status, priority` query returning counts.

## Audit logging gap on deletes

Every other mutation path in this diff logs an event:

| Operation | Audit? |
|-----------|--------|
| create_coach_task | yes |
| update_coach_task | yes |
| complete_coach_task | yes |
| **delete_coach_task** | **no** |
| create_training_plan | yes |
| update_training_plan | yes |
| **delete_training_plan** | **no** |

The delete service functions are two-liners that delegate directly to the repository without calling `AuditLogService`. This means a coach can delete tasks with no trace in the audit log. This is particularly concerning because the UI already guards deletes with a confirmation dialog, suggesting the author considers them significant operations.

## Audit event type naming inconsistency

Pre-existing tournament event types:
```python
TOURNAMENT_CREATED = "tournament_created"   # lowercase
TOURNAMENT_UPDATED = "tournament_updated"
```

New coach event types:
```python
COACH_TASK_CREATED = "COACH_TASK_CREATED"   # UPPER_CASE
COACH_TASK_UPDATED = "COACH_TASK_UPDATED"
```

The `AUDIT_EVENT_LABELS` dict in `labels.py` uses these exact string values as keys. Any future query filtering or grouping that applies case normalization will conflate or miss events. The labels dict happens to use the correct mixed casing today, so display works, but this is a maintenance landmine.

## CoachTaskDialog always emits today's date as due_date

In `CoachTaskDialog.__init__`:
```python
self.due_date_edit.setDate(QDate.currentDate())
self.due_date_edit.setSpecialValueText(" ")
```

The `setSpecialValueText` call only affects display when the date equals the minimum date, not when it equals today. `form_data()` then unconditionally converts:
```python
due_date_val = self.due_date_edit.date().toString("yyyy-MM-dd")
```

There is no mechanism to detect "user didn't touch the date" vs "user explicitly picked today." Every task created via the dialog will have a due_date set, making the nullable-due-date column in the schema unreachable from the UI. A checkbox ("Указать срок") or setting the QDateEdit to its minimum value to trigger the special-value path would fix this.

## Orphan attachment files on delete

`AttachmentsWidget._on_delete` calls `delete_attachment(connection, att.id)` which only removes the DB row:

```python
def delete_attachment(connection: sqlite3.Connection, attachment_id: int) -> None:
    connection.execute("DELETE FROM attachments WHERE id = ?", (attachment_id,))
    connection.commit()
```

The physical file under `profile_root/attachments/<entity_type>/<entity_id>/` remains on disk. The service module documents orphan *records* (when an entity is deleted) but this is about orphan *files* (when an attachment record is explicitly deleted by the user).

## Scope of the diff: two features in one branch

This branch ships Coach Workflow v1 and UI/UX Customization (P2) in a single diff. The two features share no data-layer coupling—tags, attachments, custom fields, theme, and settings are independent of coach_tasks and training_plans. Reviewing and reverting are harder when unrelated features share a branch. This is noted for process awareness, not as a blocking concern.
