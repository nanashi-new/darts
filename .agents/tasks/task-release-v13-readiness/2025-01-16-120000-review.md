# v1.3 release readiness: code cleanup and documentation update

This change prepares the v1.3 release by removing dead code (6 unused imports across 5 files plus one dead re-export module) and updating the full documentation suite—release notes, FAQ, SECURITY.md, roadmap, release checklist, current-state doc, and dev log. **Watch for:** The SECURITY.md vulnerability reporting process directs users to public GitHub Issues rather than private disclosure (confirmed). Release notes for v1.2 and v1.3 list features at summary level without "Known Issues" or migration notes sections (confirmed).

The documentation forms a coherent release narrative: roadmap marks v1.1–v1.3 done and introduces a v1.4 bucket, release notes describe the two intermediate versions, and the FAQ covers features added since v1.1. SECURITY.md replaces the default GitHub template with an application-specific policy, but its reporting mechanism uses public Issues with no private disclosure channel—fine for a local-only desktop app, though worth a conscious decision if the project ever gains a network surface.

The release checklist adds v1.3 gates that are all pre-checked, serving as post-hoc attestation. The cited test count ("151+ passed") is lower than the authoritative current-state figure ("154 passed"), suggesting different run contexts—not blocking, but a reader might wonder.

## SECURITY.md: reporting channel design

The reporting section directs users to create a public GitHub Issue for security or data-integrity problems. For a desktop app with no network surface there's nothing to exploit remotely. However, a data-integrity bug triggered by a crafted `.sqlite` import file would be disclosed publicly before a fix ships. If the project adds import-from-URL or any networked feature, this policy should be revisited with a private disclosure path (confirmed: no private channel exists currently).

## Release notes gaps

The v1.2 and v1.3 release note sections list features but omit sections that users upgrading from v1.1 would need: whether the database schema migrates automatically, whether old profiles need manual conversion, and whether any behavior changed for existing features. The current-state doc mentions restore points are created "before dangerous operations," which implies schema changes are handled, but the release notes don't surface this to end users (confirmed: no migration guidance present in the release notes).

## Roadmap duplication

The identical preamble "Идеи попадают сюда только после фиксации в `planning/00_PRIORITY.md` или отдельном task-файле." appears in both v1.4 and "После v1.4." One instance would suffice (confirmed).
